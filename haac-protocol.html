Hiroshima AI Assurance Protocol
— HAAP —

A common framework for verifiable AI adoption

広島AIアシュアランス・プロトコル — AI判断の検証可能な採用のための共通枠組み

Phase 1 Implementation PoC Complete · Logistics Profile
01 · Why HAAP
The gap

The missing layer for
verifiable AI adoption.

Existing protocols
A2A Protocol
Defines discovery, communication, and task management between AI agents. Operates at the application layer — not at transport.
MCP (Model Context Protocol)
Standardizes AI connections to tools and data sources. Focuses on capability and data access — not adoption gating.
HTTP · gRPC · Message Queues
Handle transport and delivery. Provide no semantics for adoption conditions or evidence verification.
The unanswered question
"Can this AI decision be formally adopted for real-world operations — with conditions verified against external evidence, not just claimed?"
We are not aware of a widely adopted common protocol that combines organization-defined adoption requirements, externally checked evidence, adoption-state control, and replayable decision records across organizations. HAAP is designed to address that gap.
HAAP is carried over HTTP or gRPC and can be integrated into A2A- or MCP-based workflows. It is not a replacement for those protocols — it is an accountability layer operating alongside them. A2A and MCP belong in the Agent & Tool Integration layer above transport; HAAP occupies the common accountability layer between them and business profiles.
02 · Architecture
Where HAAP sits

One common layer,
many business profiles.

Business-specific rules — temperature thresholds, logistics routes, approval authorities — live in profiles. Transport and agent protocols operate below. HAAP is the common accountability layer between them.

Business Profiles Logistics · Pharma · Manufacturing · Finance · AI Agent Delegation · …
Hiroshima AI Assurance Protocol ← This layer Requirement · Evidence · Verification · Adoption State · Authorized Transition · Record
Agent & Tool Integration A2A · MCP · Business APIs
Identity & Trust Services Signatures · Timestamps · Revocation · ADIC
Transport & Bindings HTTPS · gRPC · Message Queue

Temperature thresholds, logistics routes, and domain-specific conditions are not in the common protocol. They are defined in business profiles. HAAP provides only the common structure for fixing, verifying, and reproducing those rules — independently of what any single party asserts. The same protocol can carry a Logistics Profile, a Pharma Profile, or an AI Agent Delegation Profile.

03 · Participants
Six logical roles

One organization may hold
multiple roles.

HAAP distinguishes six logical roles. A single organization may fill more than one, but each role must be logically separated. This is consistent with the RATS architecture (RFC 9334), which separates Attester, Verifier, and Relying Party functions.

Role What they do Example
Operation Requester Defines what is to be executed — conditions for adoption, required evidence, selection window, and validity period. Shipper, buyer
Accountability Principal Seeks to execute. Submits inspection claims about its ability to satisfy conditions. AI agent, carrier, robot
Evidence Issuer Issues evidence. Each business profile defines whether the issuer must be independent of the principal and which trust relationships are acceptable. Temperature logger, certifier
Verifier Re-calculates or cross-checks claims against evidence. Produces a verification outcome. ADIC verification engine
State Transition Controller Decides whether the state change may occur based on the verification outcome. Handoff gate, approval workflow
Auditor Re-verifies packets, evidence, rules, and records after the fact. Shipper, regulator, third party

In the RATS framework (RFC 9334), the central roles are Attester, Verifier, and Relying Party. Evidence is not a role but an artifact the Attester generates and the Verifier evaluates. HAAP maps to this architecture: the Accountability Principal acts as Attester, the Verifier role maps directly, and the State Transition Controller acts as Relying Party.

04 · Message Types
Nine proposed message types

Core functions of 5 categories
validated in Phase 1.

Common Envelope — All messages are proposed to share a common envelope including: protocol_version, message_type, message_id, process_id, issuer, subject, audience, timestamps, nonce, idempotency_key, profile_id, profile_version, profile_digest, payload_digest, previous_message_digest, and proof. These fields are designed to support issuer identification, profile binding, chain linkage, and replay protection. These properties depend on signature verification, trust anchor configuration, canonicalization scheme (e.g. RFC 8785), and verifier implementation — none of which are specified in this draft.
Wire format (JSON Schema, Content-Type, required vs optional fields), signing algorithms, and error handling are not yet specified. This overview describes intended message semantics only.

05 · State Model
Four state types

Evidence shortage and
condition failure kept separate.

HAAP proposes four distinct state categories. Conflating processing progress, verification outcomes, and authorization decisions leads to ambiguity in audit and exception handling. The fourth category — Accountability State in the Responsibility OS sense — describes who holds what responsibility and what remains unconfirmed; this is preserved in the accountability-relevant information packet and is distinct from the other three.

Accountability State in the Responsibility OS glossary sense — who confirmed what, who is responsible, what conditions were unverified — is preserved inside the accountability-relevant information packet itself and is distinct from these four operational state categories. This is the "responsibility state" referenced in the LP glossary section.

06 · Phase 1 PoC
Implementation PoC results

Logistics Profile:
What Phase 1 validated.

Phase 1 validated core functions corresponding to five of the nine proposed message categories in a single domain — pharmaceutical cold-chain logistics in Hiroshima. This is not a proof of the full HAAP protocol. It is an implementation PoC of the core verification and chain-stop engine in one business profile.

開発元：株式会社オンザリンクス（共同特許出願人） · 製薬コールドチェーン引き渡し業務を対象とした公開実装PoC

Validated in Phase 1
Business rule and ruleset binding — profile_id, profile_version, profile_digest, ruleset_digest fixed at evaluation time
Evidence field inspection — gap detection, consistency checks across seal, custody, logger, temperature records
Independent metric recalculation — d+, d−, a+, a−, gap, duration recalculated from temperature records
Verdict generation — RELEASE / HOLD / QA_REVIEW / REJECT with alpha/beta two-stage thresholds
Downstream chain-stop control — halt on non-RELEASE, last_good / first_fail anchors recorded
Digest and replay verification — same implementation, same input produces same output
Not yet validated
Evidence-source authenticity — sensor and logger integrity not verified
Production system integration with 株式会社オンザリンクス operational environment
Cross-company API exchange — inter-organizational message passing
Multi-implementation interoperability — single implementation only in Phase 1
Multi-candidate selection and selection proof — Phase 2 target
Authorization message format — wire format, nonce, idempotency key not implemented
Regulatory compliance and SLA — outside scope
Phase 1 Verdict Verification Outcome (B) Adoption State (C) Transition Authorization (D)
✅ RELEASE → SATISFIED → ADOPTABLE → AUTHORIZED
🟡 HOLD → INDETERMINATE → PENDING_EVIDENCE → NOT_AUTHORIZED
🟠 QA_REVIEW → SATISFIED* → CONDITIONALLY_ADOPTABLE → REQUIRES_REVIEW
🔴 REJECT → UNSATISFIED → NOT_ADOPTABLE → NOT_AUTHORIZED

* QA_REVIEW indicates alpha-threshold crossing without beta-threshold violation — conditions partially satisfied, pending human review. Chain-stop: when any upstream segment returns NOT_AUTHORIZED, downstream evaluations halt. last_good and first_fail anchors are recorded in the append-only log without modifying prior entries.
